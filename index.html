Agile vs Waterfall

Throughout my school life (GCSE’s / Sixth Form) I learnt a great deal around traditional project management (waterfall) the projects were typically structured with a great deal of emphasis around documentation and a large amount of up-front planning with little leniency to change further down the line. I became confident conducting projects within a waterfall like manner and became used to large amounts of pre-planning such as drafting requirement specs and producing large amounts of testing documentation.

However, throughout the second year of university I was introduced to the agile methodology, this specifically being scrum. Our second-year project attempted to replicate a scrum team by using methods such as the concept of sprints, burndown charts and story points.

In my current development role, we stick to agile principals, operate the usual ceremonies and have an agile mindset towards our work. I am comfortable running / leading ceremonies and will often be an influencer within these sessions. I enjoy the concept of 2 week sprints as it allows us to pivot quickly, introduce new technologies with minimal lead time and the ability to fail fast without a massive sunk cost.

I have attempted to introduce new concepts to the team such as regular cascade sessions / show and tell sessions. We use Agile Central (Rally) to manage our backlog and prioritise items and I am comfortable using this in day to day work. I also use trello for managing my personal development backlog, and am looking forward to dive into it more.

Java Chat Application

This program was developed as part of my second year Operating Systems and Networking module. It was developed to help represent the way operating systems use networks to communicate to each other. The software uses Java socket connections to connect an unlimited number of clients to a listening server that is multi-threaded and uses a thread pool handler to delegate a new thread to each new connection that comes in. The program also has built in exception handling to deal with connection drops and user errors. There are a select number of commands that the user can send to the server, which has a hash map coded message board, where messages can be sent to and read from.

The program worked well with quite a few test cases however could have been improved with slightly more validation such as Regex and a GUI could be used with the foundations that have been set in place to deliver an attractive functioning chat messaging client and server.

The source code can be viewed via Github.

Java 3D Train Scene

The 3D project for my university second year Multimedia and Computer Graphics module was to design a 3D train scene using the Java3D framework. This framework uses the notion of scene graphs for graphics manipulation and was an interesting way of designing a scene. The code was manually typed in as Java3D does not have any GUI tools to generate the objects, properties, geometries and transformations used in the program.

Furthermore, basic lighting was applied to the geometric shapes and different textures were applied to the objects such as matte colours. Rotation interpolators were also used as part of the Java3D framework which allowed for animation of the wheels by specifying such dimensions as rotation angle and time taken. Some basic mathematics was also used in the program as translations had to be defined in terms of vectors and π for the rotation of objects.

This program was an interesting insight into designing graphics from scratch as opposed to using a modern GUI tool to automate the process, however there was a lack of documentation online for various functions of Java3D, which made the programming a tad more difficult than it should have been. I would prefer to work with a more modern and better documented framework in the future.
